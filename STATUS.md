## 🔴 核心瓶颈：变声拦截需要 Root

**这是整个项目最大的卡点。**

### 问题本质

Android 的音频系统（AudioFlinger）在没有 root 权限时，应用无法直接拦截系统级音频流并注入处理后的音频。当前三种桥接方案都有各自的局限：

| 方案 | 权限 | 能否拦截系统音频 | 局限 |
|------|------|:---:|------|
| **Root AudioFlinger** | Magisk root | ✅ 能 | 需要 root，普通用户无法使用 |
| **Shizuku AAudio** | Shizuku | ❌ 不能 | 只能处理 App 自身的音频，无法作为系统虚拟麦克风 |
| **无障碍服务** | 无障碍权限 | ❌ 不能 | 最兼容但同样无法拦截其他 App 的音频 |

### 为什么这么难

- Android 从 10 开始收紧了 `AudioRecord` 的权限，非 root App 无法录制其他 App 的音频
- `AudioPlaybackCapture`（Android 10+）理论上可以录制其他 App 音频，但需要目标 App 明确声明 `allowPlaybackCapture=true`，且**不能作为麦克风输入**
- 虚拟麦克风的终极目标是：**任何 App 都能用 MaidMic 作为麦克风输入源**（游戏语音、直播、视频通话），但 Android 没有开放"注册为音频输入设备"的非 root 路径
- `MediaProjection`（屏幕录制 API）可以捕获音频，但它是**输出流**，不能反向注入到输入设备

### 正在探索的方向

1. **Shizuku + 修改 AudioPolicy** — 通过 Shizuku 调用系统服务，尝试注入虚拟音频设备（不稳定，依赖厂商实现）
2. **无障碍 + 通知栏转发** — 用无障碍服务监听通知，配合悬浮窗实现"伪虚拟麦克风"（用户体验差）
3. **等待 Android 新版本** — Android 15/16 是否会开放虚拟音频设备 API（未知）
4. **社区方案调研** — 参考 WO Mic、AudioRelay 等已有的虚拟麦克风 App 如何实现（大多也需要 root 或 PC 端配合）

## 🔴 其他槽点

### 1. DSP 延迟不稳定
- 简单处理管线（增益+EQ）约 5-10ms，可接受
- 完整处理管线（增益+EQ+变调+混响+共振峰+失真+回声）延迟可能超过 50ms
- 实时变声场景下 >50ms 就有明显感知，>100ms 基本不可用
- **需要优化**：JNI 层流水线并行化、buffer size 动态调整

### 2. 变调后人声不自然
- 当前变调用的是简单的重采样（线性插值），改变音高的同时也改变了共振峰
- 虽然加了 Formant Shifting 补偿，但效果有限，特别是大幅变调（±8 半音以上）时很假
- **需要**：更好的 pitch shifting 算法（如 PSOLA 或 phase vocoder）

### 3. 没有 Auto-Tune
- 用户期望的"变声"不只是粗暴变调，还有自动修音（Auto-Tune）
- 实现需要实时基频检测（YIN/pYIN 算法）+ 音阶量化 + 平滑过渡
- 计算量大，对实时性要求高

### 4. P2P 传输不够稳定
- 5月1日新加的二维码连接 + WiFi/蓝牙双传输
- WiFi 模式受路由器和网络环境影响大
- 蓝牙模式带宽有限，高采样率下可能丢包

### 5. UI 有待打磨
- Canvas 频响曲线图在低性能设备上可能卡顿
- 滑块交互不够精细，微调困难
- 预设切换时偶有爆音
